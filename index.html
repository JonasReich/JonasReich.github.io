Visualizing the Telekinesis Beam

The telekinesis spell, used to lift up the energy crystals, is visualized with a beam from the player to the grabbed crystal. This beam had to be very organic - using a laser-pointer-like line felt far too stiff and undynamic.
To solve this issue, I introduced a slight "lag", so the crystal lingers at its current position for a moment and slowly approaches its destination. This movement is calculated by interpolating the current crystal position with its desired position (represented by a red line in the gif above).

To draw the actual beam, I made use of Unity's line renderer component (which was fortunately improved in version 5.5). I fueled in coordinates (displayed as white hairlines) based on a sine curve that has its x axis on the "ideal ray" from the player to the desired position. 
void DrawBeam ()
{
	lineRenderer.enabled = true;

	Vector3 idealBeam = telekinesisPointer.transform.position - transform.position;
	Debug.DrawLine(transform.position, transform.position + idealBeam);

	float stepLength = idealBeam.magnitude / lineRenderer.numPositions;
	Vector3 maxOffset = grabbedCrystal.transform.position - telekinesisPointer.transform.position; // local coordinates
	Debug.DrawLine(telekinesisPointer.transform.position, telekinesisPointer.transform.position + maxOffset);

	Vector3[] positions = new Vector3[lineRenderer.numPositions];
	for (int x = 0; x < lineRenderer.numPositions; x++)
	{
		Vector3 tempPos = transform.position + idealBeam.normalized * stepLength * x; // global coordinates
	
		float y = Mathf.Pow((x / (float)(lineRenderer.numPositions)), 2);
		Vector3 offset = maxOffset * y;
		Debug.DrawLine(tempPos, tempPos + offset);
	
		positions[x] = tempPos + offset;
	}

	lineRenderer.SetPositions(positions);
}

Flying dragons

For a long time of our development process, we wanted to include dragons in our game. They were supposed to fly in groups as certain fish swims in swarms.
To accomplish this behaviour, I wrote a simple flocking algorithm. The main movement is controlled by following a predetermined, node-based path. The final node of the path is the actual target, around which all of the dragons gather. Whenever I move the target, the agents follow, always trying to keep a distance to the target-point.
I'm curious to experiment and build upon this behaviour in the future. 
void Move()
{
	steering = Vector3.zero;
	var _position = transform.position;

	// follow a target/path
	var _targetPosition = targetTransform.position;
	var _distanceToNextTarget = Vector3.Distance(_position, _targetPosition);

	var _desiredVelocity = (_targetPosition - _position).normalized * maxSpeed;
	var _followForce = (2*_desiredVelocity + Velocity).normalized * followForceFactor;

	steering += _followForce;
	Debug.DrawLine(_position, _position + _followForce, Color.yellow);
	
	// avoide obstacles
	var _view_ahead = _position + Velocity.normalized * detectionDistance;
	var _mostThreatening = FindMostThreateningObstacle(_view_ahead, _view_ahead / 2);

	if (_mostThreatening != null)
	{
		var _avoidanceForce = _view_ahead - _mostThreatening.transform.position;
		_avoidanceForce *= avoidanceForceFactor;
		steering += _avoidanceForce;
		Debug.DrawLine(_position, _position + _avoidanceForce, Color.red);
	}

	// align with other flockagents
	if (flockAgentPool.Count > 0)
	{
		var _neighbourCount = 0;
		var _alignmentForce = Vector3.zero;
		var _cohesionForce = Vector3.zero;

		foreach (var enemy in flockAgentPool)
		{
			var _agent = enemy.FlockAgent;
			if (_agent != this && Vector3.Distance(_position, _agent.transform.position) < agentRadius)
			{
				_neighbourCount++;
				_alignmentForce += _agent.Velocity;
				_cohesionForce += _agent.transform.position;
			}
		}

		_cohesionForce /= _neighbourCount;
		_cohesionForce -= _position;

		_alignmentForce.Normalize();
		_cohesionForce.Normalize();

		var _separationForce = -2 * _cohesionForce * separationForceFactor;
		_cohesionForce *= cohesionForceFactor;
		_alignmentForce *= alignmentForceFactor;

		steering += _separationForce + _cohesionForce + _alignmentForce;
		Debug.DrawLine(_position, _position + _separationForce, Color.cyan);
	}

	// apply forces
	steering = Vector3.ClampMagnitude(steering, maxForce);
	steering /= steeringMass;

	Velocity += steering * Time.deltaTime;
	Velocity = Vector3.ClampMagnitude((Velocity), maxSpeed);
	transform.position += Velocity * Time.deltaTime;

	Debug.DrawLine(transform.position, transform.position + Velocity);

	// rotate according to Velocity
	var _rotationDelta = transform.rotation;
	_rotationDelta.SetLookRotation(_desiredVelocity);
	transform.rotation = _rotationDelta;
}

A Zen-Like Experience


The biggest difference of Aeronauts compared to traditional strategy games is the complete absence of militarized fights. There are enemy encounters on the dungeon-islands, but they are optional up to the point when you decide to engage.
This approach creates a very calm, zen-like experience that allows you to focus solely on exploration and unit-management.

One key component to achieve that goal is the resource tree which is inspired by the goods from The Settlers II. It is a relatively peaceful game compared to major strategy competitors such as Age of Empires, Civilisation or the Anno series. The game feel is already pretty close to the end goal we were aiming for. However, the economy of the Settlers II is still centered around the military, which is the driving force of every decision. All the fortunes a player collects over the course of a game are spent to resolve military disputes. Gold coins which are used to recruit and train soldiers are the most expencive good. Unfortunately there is no civil equivalent to gold coins that is worth your time or resources.

In Aeronauts, we wanted to take a different route: Our resource tree is built around cloth which is substantial for crafting tools and "breeding" new Aeronaut workers.
The final goal of the game is the construction of a big air-ship - which again requires a lot of cloth, wood and other scarce resources.
